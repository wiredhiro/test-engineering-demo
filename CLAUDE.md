# CLAUDE.md — テスト作成ルール

このリポジトリでテストを書く／レビューするときの規約。AI（Claude Code）にも人にも、この通りに書いてもらう。
抽象論でなく「こう書け／こう書くな」を具体例で示す。

## このリポジトリ

ユーザー登録機能を題材にしたテストデモ。

- **単体**：Vitest（`test/*.test.ts`）
- **結合**：Vitest ＋ better-sqlite3 のテスト専用インメモリDB（`test/*.integration.test.ts`）
- **E2E**：Playwright（`e2e/*.e2e.ts`）
- コマンド：`npm test` ／ `npm run e2e` ／ `npm run lint` ／ `npm run typecheck`

## テストを書くときのルール

1. **観点表を正とする**：まず `観点表.xlsx`／`観点表_3層.md` の観点を決めてから書く。**テスト名に観点ID（U-06 等）を入れ**、表と1対1で対応させる
2. **AAA で書く**：準備（Arrange）→ 実行（Act）→ 確認（Assert）の3ブロック。Act は原則1回
3. **FIRST を守る**：特に**再現性**。時刻・乱数・通信に依存しない
4. **期待値は独立に計算した値を直接書く（トートロジー禁止）**
   ```ts
   expect(add(2, 3)).toBe(2 * 3); // ❌ 実装から写した式
   expect(add(2, 3)).toBe(5);     // ✅ 答えの数字
   ```
5. **正常系だけでなく、異常系・境界値・null・文字も必ず入れる**（境界は「ちょうど」と「超え」を両方）
6. **モックは外部依存だけ**：外部API・DB は代役に。自分のロジックはモックしない
7. **結合テストはテスト専用の使い捨てDB**（`:memory:`）：`beforeEach` で作り直し、`afterEach` で閉じる
8. **E2E は主要動線だけ**：固定待ち（`sleep`）禁止。自動待機（`expect(...).toBeVisible()`）を使う

## 禁止事項

- `new Date()` ／ `Math.random()` の直呼び（外から渡して固定する）
- テストからの**実API・本番DB**へのアクセス
- `expect(x).toBeDefined()` だけの弱い検証（具体的な値を確認する）
- 期待値を実装からコピーする（トートロジー）

## AI生成テストをレビューするときの観点

次のアンチパターンを探す。指摘は「**観点 ＋ 理由 ＋ 代案**」の3点セットで書く。

- トートロジー（期待値を実装から写している）
- 実装に追従しただけ（仕様でなく今の動きに合わせている）
- 弱い検証（`toBeDefined` だけ 等）
- 異常系・境界値の抜け
- 過剰モック（何も検証していない）
- flaky の温床（時刻・乱数・待ち方）
- 似たテストの重複（パラメータ化でまとめる）

くり返し出る指摘は、このファイルのルールに昇格させる（レビュー → ルール化）。
