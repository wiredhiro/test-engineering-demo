# 観点表（単体・結合・E2E）

ユーザー登録という1つの機能を、3つの層で確認するときの観点表。
それぞれ、実際のテストコードと1対1で対応している（IDがテスト名に入っている）。

## 大事な考え方：同じ機能でも、層ごとに「見る観点」を分ける

同じ登録機能でも、全部の観点を全部の層で確認するのではない。**下の層で確認できることは下で確認し、上の層は上でしか見えないことに絞る**（テストピラミッド）。

| 層 | この層でしか見えないこと | 観点の数 |
|---|---|---|
| 単体 | 入力チェックの細かい分岐（境界値・文字種） | 多い |
| 結合 | DBとのつなぎ目（保存・重複・状態） | 中 |
| E2E | 画面に結果が正しく出るか（ユーザーの体験） | 少ない（主要動線だけ） |

---

## ① 単体テストの観点表 — `validateUsername`

入力チェックの部品ひとつ。**細かい分岐はここで網羅する**（正常・異常・境界・null・文字）。
対応：`test/validateUsername.test.ts`

| ID | 確認する観点 | 分類 | 入力 | 期待する結果 |
|---|---|---|---|---|
| U-01 | 有効な名前は通る | 正常系 | `"tanaka_9"` | OK |
| U-02 | 空文字を弾く | 異常系 | `""` | NG「入力してください」 |
| U-03 | null を弾く | null | `null` | NG「入力してください」 |
| U-04 | 下限ちょうど(3文字)は通る | 境界値 | `"abc"` | OK |
| U-05 | 下限未満(2文字)を弾く | 境界値 | `"ab"` | NG「3文字以上」 |
| U-06 | 上限ちょうど(20文字)は通る | 境界値 | `"a"×20` | OK |
| U-07 | 上限超え(21文字)を弾く | 境界値 | `"a"×21` | NG「20文字以内」 |
| U-08 | 記号を含むと弾く | 文字 | `"tanaka!"` | NG「半角英数字と _」 |
| U-09 | 全角を含むと弾く | 文字 | `"たなか"` | NG「半角英数字と _」 |
| U-10 | 空白を含むと弾く | 文字 | `"tan aka"` | NG「半角英数字と _」 |

## ② 結合テストの観点表 — `registerUser` ＋ DB

バリデーション〜保存のつなぎ目。**本物のDBを使わないと見えない観点**に絞る（保存・重複・状態）。
入力チェックの細かい分岐（U-05〜U-10）はここでは繰り返さない。単体でカバー済みだから。
対応：`test/registerUser.integration.test.ts`（テスト専用の使い捨てDB）

| ID | 確認する観点 | 分類 | 前提 → 操作 | 期待する結果 |
|---|---|---|---|---|
| I-01 | 各テストは空のDBから始まる | 独立性/前処理 | まっさらなDB | 件数 0 |
| I-02 | 新規登録が実際にDBに入る | 正常系/つなぎ目 | 空 → `"tanaka_9"` を登録 | OK＋DBに1件ある |
| I-03 | 同じ名前は二重登録できない | 状態 | 既に `"tanaka_9"` あり → もう一度登録 | NG「使われています」 |
| I-04 | 二重登録してもDBは1件のまま | 状態 | 同上 | 件数 1 |
| I-05 | バリデーションNGは保存されない | 異常系/つなぎ目 | 空 → `"ab"`(短すぎ)を登録 | NG＋件数 0（①で止まる） |

## ③ E2Eテストの観点表 — 登録フォーム画面

画面〜サーバー〜DB〜画面表示の全部を、ブラウザ操作で通す。**数を絞り、ユーザーの主要動線だけ**。
細かい入力パターンは単体・結合に任せ、E2Eは「押したら結果が画面に出るか」を見る。
対応：`e2e/signup.e2e.ts`（Playwright）

| ID | 確認する観点 | 分類 | 画面での操作 | 期待する表示 |
|---|---|---|---|---|
| E-01 | 登録が成功する | 正常系（主要動線） | 有効な名前を入力 → 登録 | 「登録しました」 |
| E-02 | 入力エラーが画面に出る | 異常系 | 短すぎる名前 → 登録 | 「3文字以上で入力してください」 |
| E-03 | 二重登録が画面で防がれる | 状態 | 同じ名前で2回登録 | 「その名前は使われています」 |

### E2Eで「あえて実装しない」観点

次のような観点も洗い出せるが、**E2Eには入れない**。単体・結合で既に確認しているから（数を絞るのがE2Eの原則）。

| 観点 | どこでカバー済みか |
|---|---|
| 空入力・null | 単体 U-02, U-03 |
| 境界値（3文字・20文字ちょうど） | 単体 U-04〜U-07 |
| 記号・全角・空白 | 単体 U-08〜U-10 |
| DBに実際に保存されたか | 結合 I-02 |

## この観点表の使い方

- **設計の順番**：まずこの表を書いてから、テストコードにする（いきなりコードを書かない）
- **層の分担**：ひとつの観点は、一番下の層で確認する。上の層で重複させない
- **AIへの渡し方**：「この観点表の I-03 をテストにして」と Claude Code に渡せる形になっている

---

## テストの後：実施結果を記録し、フィードバックする

観点表は「設計図」で終わりではない。**テストを実施したら、その結果を表に書き戻す**。
ここで観点表は、設計図から **実施記録** へと役割を変える（「観点表の作成・**整備**」「ナレッジ化」がこれ）。

### ① 実施結果を、同じ行に記録する（統合・1タブ）

観点表と不具合票は、**別々の表にしない**。失敗したら、その観点の**行の右側に「実施」列と「不具合」列を足す**。
1行＝1観点で完結し、1つのタブで管理できる（Excel の実物が `観点表.xlsx`）。1行は3ブロックになる：

- **【設計】** ID・層・観点・分類・入力・期待する結果・優先度
- **【実施】** 実施結果（合格／失敗）・実施日
- **【不具合】**（失敗した行だけ記入）実際・再現手順・重要度・状態・Issue番号・対応日

例（合格の行は不具合欄が空欄、失敗した U-06 だけ埋まる）：

| ID | 期待する結果 | 実施結果 | 実際（失敗時） | 状態 | Issue |
|---|---|---|---|---|---|
| U-07 | NG「20文字以内」 | ✓ 合格 | —（空欄） | — | — |
| U-06 | OK（20文字ちょうど） | ✗ 失敗 | 「20文字以内」エラーが出る | 修正済 | #12 |

（実物はもっと列があるが、考え方はこの通り。合格行は不具合の列が空欄なだけ）

これで「どの観点が・いつ・通ったか／落ちたか」も「その失敗が今どうなっているか」も、**同じ行**で分かる。
CI（自動実行）を使う場合も、落ちた観点の ID をそのまま報告できる。

### ② 結果をフィードバックして、観点表を育てる

記録した結果は、次の3つに返す。**このループが「整備」の中身**：

1. **落ちた観点** → 不具合として登録し、直したら再テスト（合否欄を更新）
2. **想定外に見つかったバグ** → その原因を **新しい観点として表に追加**する（次からは必ず確認される）
3. **抜けていた視点** → チームで共有し、観点の「切り口」を更新する（ナレッジ化）

> **具体例**：もし「絵文字を含む名前」で本番バグが出たら、それは観点表に無かった視点。
> `U-11 絵文字を含むと弾く（文字）` を表に足す。以後、この観点は毎回確認される。
> —— こうして観点表は、バグを経験するたびに賢くなっていく。

この「結果 → 観点表に反映 → 次に活かす」は、レビューの指摘を CLAUDE.md に昇格させる
**レビュー → ルール化**（ガイド §6）と同じ発想。観点表もルールファイルも、**育てるもの**。

## 不具合管理：見つけたバグをどう回すか

観点表で「✗ 失敗」が出たら、黙って直さず、**その行に不具合を記録して、閉じるまで追う**。
別の「不具合票」は作らない——**失敗した観点の行に、不具合の列を足すだけ**（統合・1タブ）。

### 失敗した行に足す「不具合」の列

| 列 | 例 |
|---|---|
| 実際（どう失敗したか） | 20文字ちょうどが登録できない |
| 再現手順 | 20文字の名前で登録 → 弾かれる |
| 重要度 | 中 |
| 状態 | 修正済 |
| Issue番号 | #12（GitHub 連携用・任意） |
| 対応・再テスト日 | 07/05 |

「期待」は設計の**「期待する結果」列を再利用**するので足さない。「実際」だけ書けば足りる。
そして「再現手順」が特に大事——**開発者が手元で同じ現象を出せる**ように書く（出せないと直せない）。

### 状態の流れ（ここを最後まで追うのがテスターの仕事）

```
新規(Open) → 修正中 → 修正済(Fixed) → 再テスト ─ OK → 完了(Closed)
                                          └ NG → 再オープン(Reopen)
```

ポイントは、**「バグを見つけて登録して終わり」ではない**こと。
修正後に**もう一度テストして（再テスト）**、落ちていた観点（U-06）の実施結果欄を「✓ 合格」に
更新して、はじめて完了。ここまでがワンセット。

### 別シートは要らない：「失敗」でフィルタすれば不具合一覧になる

「一箇所にまとめたい」の答えがこれ。不具合は同じ行に入っているので、
**実施結果を「失敗」でフィルタする**だけで、画面がそのまま**不具合一覧**になる。

- 不具合票の別シートを持たなくていい（**二重管理が起きない**）
- 「失敗だけ」「単体だけ」「未完了の状態だけ」など、見たい切り口に1クリックで絞れる
- Excel の実物 `観点表.xlsx` に、この仕組み（プルダウン・失敗行の自動強調・フィルタ）を入れてある

### GitHub で回すなら

- **不具合1件 = GitHub Issue 1件**。Issue 番号がそのままバグ ID（`#12` ＝ BUG-12）
- 観点表の不具合欄に `#12`、Issue 側に「関連観点：U-06」を書いて**相互リンク**
- 直す PR の説明に `Closes #12` と書くと、マージで Issue が自動クローズ
- CI（自動テスト）が緑に戻れば、**再テストOKの証拠**がそのまま残る

この不具合票を、GitHub の**入力フォーム**として現物にしたのが `.github/ISSUE_TEMPLATE/bug_report.yml`。
「現象・期待/実際・再現手順・重要度・優先度」を必須にしてあり、`関連観点ID` 欄で観点表とつながる。

### 重要度と優先度は分けて書く

- **重要度（どれくらい深刻か）**：データが消える＝高、見た目のズレ＝低
- **優先度（どれくらい急ぐか）**：よく使う画面のバグ＝先に直す

分けておくと「深刻だが滅多に起きない」バグの扱いを、チームで判断できる。

### 観点表とのつながり（一周させる）

```
テスト実施 → ✗失敗 → 不具合IDを発行(BUG-12) → 修正 → 再テスト
   → 観点表の実施結果を✓に更新 →（想定外なら新しい観点も追加）
```

この一周で、観点表は「設計図」から「実施記録」になり、さらに「次に活きるナレッジ」へと育つ。
