# サンプル資料：ユーザー名バリデーションのテスト

テスト設計の進め方を、小さな1関数で示す資料。「観点を洗い出して → テストにして → バグを捕まえる」までの流れを、実際に動くコードで見せる。

対象：`validateUsername(name)` … ユーザー名が使えるか判定する関数（3〜20文字・半角英数字と `_`）。

---

## 作った順番（設計 → 実装 → レビュー）

### ① 設計：先に観点表を書く

いきなりコードを書かず、まず「何を確認すべきか」を洗い出す（この資料の後半＝観点表）。
正常系・異常系・境界値・null・文字の視点から、10個の確認項目（U-01〜U-10）を出している。

### ② 実装：観点表をテストにする

観点表の各行を、そのままテストコードにする。各テストの名前に観点表の ID（U-06 など）を
入れてあるので、表とコードが1対1で対応する。実行すると全10件が緑（成功）になる。

```
 ✓ validateUsername.test.ts (10 tests) 9ms

 Test Files  1 passed (1)
      Tests  10 passed (10)
```

### ③ レビューの実演：バグを1個仕込んで、テストが捕まえるのを見せる

「テストがちゃんと仕事をする」ことを見せるため、わざと **境界のバグ** を仕込む。
上限チェックを `name.length > 20`（正）から `name.length >= 20`（誤）に1文字だけ変える。
これで「20文字ちょうど」が誤って弾かれるようになる（off-by-one＝1つのズレ、という定番バグ）。

すると、境界値の観点として置いておいた **U-06（上限ちょうどは通る）だけが赤くなる**：

```
 ❯ validateUsername.test.ts (10 tests | 1 failed)
   × validateUsername > U-06 上限ちょうど(20文字)は通る
     → expected { ok: false, error: '20文字以内で入力してください' }
       to deeply equal { ok: true }

      Tests  1 failed | 9 passed (10)
```

**ここが一番のポイント**：もし観点表で「上限ちょうど」を確認していなければ、この1文字のバグは
すり抜けていた。**境界値の観点を先に洗い出していたから、バグを捕まえられた**（バグを戻せば再び全部緑になる）。

---

## この資料で伝えたいこと

- テストは「書ける」だけでなく、**何を確認すべきかを先に設計してから書く**
- 特に **境界値**（以上・以下の境目）はバグの巣。だから観点として必ず置く
- テストの価値は、こうして **バグを実際に捕まえるところ** にある

---

# 観点表：`validateUsername`

テストコードを書く前に、「何を確認すべきか」を先に洗い出したもの（上記①）。
テストコードは、この表を1行ずつ翻訳したもの。

## 仕様

ユーザー名が使えるかどうかを判定する。

- 空 / null は不可
- 3文字以上、20文字以内
- 使えるのは半角英数字と `_` のみ

## 観点表

| ID | 確認する観点 | 分類 | 入力 | 期待する結果 |
|---|---|---|---|---|
| U-01 | 有効な名前は通る | 正常系 | `"tanaka_9"` | OK |
| U-02 | 空文字を弾く | 異常系 | `""` | NG「入力してください」 |
| U-03 | null を弾く | null | `null` | NG「入力してください」 |
| U-04 | 下限ちょうど(3文字)は通る | 境界値 | `"abc"` | OK |
| U-05 | 下限未満(2文字)を弾く | 境界値 | `"ab"` | NG「3文字以上」 |
| U-06 | 上限ちょうど(20文字)は通る | 境界値 | `"a"×20` | OK |
| U-07 | 上限超え(21文字)を弾く | 境界値 | `"a"×21` | NG「20文字以内」 |
| U-08 | 記号を含むと弾く | 文字 | `"tanaka!"` | NG「半角英数字と _」 |
| U-09 | 全角を含むと弾く | 文字 | `"たなか"` | NG「半角英数字と _」 |
| U-10 | 空白を含むと弾く | 文字 | `"tan aka"` | NG「半角英数字と _」 |

## この表の狙い

「観点の切り口」を、この機能で具体化したもの。
**正常系 / 異常系 / 境界値 / null / 文字** の各視点から、抜けなく代表を選んでいる。

特に **境界値（U-04〜U-07）** が要。「以上・以下」を1文字間違える（off-by-one）バグは、
まさにこの境目でしか出ない。上限ちょうど(U-06)と上限超え(U-07)を両方置くことで、
`> 20` を `>= 20` と書き間違えるバグを捕まえられる（→ 上記「③ バグ実演」）。

# 結合テスト：テスト専用DBを使う

ここまでは単体テスト（部品ひとつ）。ここからは **結合テスト**＝本物のDBを使って、
複数の部品の「つなぎ目」まで確認する。

対象：`registerUser(name, db)` … ユーザー登録。次の3つの部品が連なる。

```
入力 → ① バリデーション → ② DBで重複チェック → ③ DBに保存 → 完了
```

①だけを調べるのが単体テスト（前半の validateUsername）。①〜③のつなぎ目を本物のDBで
調べるのが結合テスト。

## 単体テストとの違い

| | 単体（validateUsername） | 結合（registerUser） |
|---|---|---|
| 調べる範囲 | 部品ひとつ（入力チェック） | ①〜③のつなぎ目 |
| DB | 使わない | **本物のDBを使う** |
| 例：確認できること | 「2文字は短すぎ」 | 「同じ名前は二重登録できない」 |

「二重登録できない」は、バリデーション単体では絶対に確認できない。
**"DBに既にある"という状態と、保存処理のつなぎ目**を見て初めて分かる。これが結合テストの守備範囲。

## テスト専用DBの使い方（結合テストの肝）

結合テストは、本番でも普段の開発用でもなく、**テスト専用の使い捨てDB**を使う。

- **本番DBは絶対に使わない** … テストはデータを消したり書いたりするので、本番が壊れる
- **普段の開発用DBも共用しない** … 残っているデータでテストが不安定になる（＝flaky）
- 代わりに、**テストごとに作り直せるメモリ上のSQLite**を使う（`":memory:"`）

コードでは、この2つが要（前処理と後始末）：

```ts
beforeEach(() => {
  db = 新しいメモリ上のDB();     // 各テストの前に、まっさらな使い捨てDBを新しく作る
  テーブルを用意する();          // CREATE TABLE users ...
});

afterEach(() => {
  db.close();                   // 後始末。次のテストにデータを残さない
});
```

これで **各テストが「空のDB」から始まる**。どのテストから走っても結果が同じ＝順序に依存しない、
安定したテストになる。

## 4つの結合テスト

| # | 確認していること | つなぎ目のどこを見るか |
|---|---|---|
| 1 | 各テストは空のDBから始まる | 使い捨てDBが毎回リセットされる |
| 2 | 新規登録が実際にDBに入る | ③保存が本当に効いたか（DB側から確認） |
| 3 | 同じ名前は二重登録できない | ②重複チェック → 保存を止める |
| 4 | バリデーションNGはDBに1件も入らない | ①で止まり、③まで進まない |

実行すると、単体10件と合わせて全14件が緑になる。

```
 ✓ validateUsername.test.ts        (10 tests)
 ✓ registerUser.integration.test.ts (4 tests)

      Tests  14 passed (14)
```

# E2Eテスト：本物のブラウザで操作する

単体・結合は、どちらも **画面を通っていない**。最後は **E2E（End-to-End＝端から端まで）**。
本物のブラウザを動かし、ユーザーと同じ操作（入力・クリック）で、画面〜DB〜画面表示の全部を通す。

対象：登録フォームのある画面（`/signup`）。

## 何が「端から端まで」なのか

E2E のテストはこう書く：

```ts
await page.goto("/signup");                                // 画面を開く
await page.fill('input[name="username"]', "tanaka_9");     // 入力欄に打つ
await page.click('button[type="submit"]');                 // ボタンを押す
await expect(page.getByText("登録しました")).toBeVisible(); // 画面の結果を見る
```

このとき実際に動くのは、全部の層：

```
ブラウザ → フォーム入力 → 通信 → サーバー → registerUser → DB（SQLite）
        → レスポンス → 画面に「登録しました」を表示
```

「ボタンを押したら成功メッセージが出る」という、**ユーザーが実際に見る動き**を確認できるのは E2E だけ。

## 3本のE2Eテスト

| # | 確認していること |
|---|---|
| 1 | ユーザー名を入力して登録すると、成功メッセージが出る |
| 2 | 短すぎる名前は、エラーメッセージが出る |
| 3 | 同じ名前は二重登録できない（画面操作を2回して確認） |

## E2E も「テストファイル」を作る（＋設定とアプリ）

E2E も、単体・結合と同じく**テストファイルを作る**。ただし違いが2つ：

| | 単体テスト | E2E |
|---|---|---|
| ファイル・コマンド | `*.test.ts` ／ `npm test` | `*.e2e.ts` ／ `npm run e2e` |
| 必要なもの | テストファイル1個で完結 | テストファイル ＋ **設定 ＋ 動くアプリ** |

E2E は「動くアプリ」をブラウザで操作するので、単体のようにファイル1個では済まない。
**テストファイル（操作を書く）＋ 設定（アプリを自動起動）＋ アプリ本体**の3点セットになる。

## E2Eの勘所

- **数を絞る**：遅く・壊れやすいので、主要動線だけ（テストピラミッドの頂点）
- **待ちは自動待機に任せる**：`toBeVisible()` は「出るまで自動で待つ」。固定待ち（sleep）は flaky の元なので書かない
- **テスト専用の環境**：アプリはテスト専用の使い捨てDBにつなぐ。本番は叩かない

---

# この資料の結び：3層の使い分け

| 層 | 対象 | 通る範囲 | 速さ・数 |
|---|---|---|---|
| **単体** | 関数ひとつ（validateUsername） | 部品ひとつ | 速い・多く |
| **結合** | 関数＋DB（registerUser） | 部品のつなぎ目 | 中 |
| **E2E** | 画面＋サーバー＋DB | 端から端まで | 遅い・少なく |

- **下（単体）を厚く、上（E2E）を薄く**積むのが良いバランス（テストピラミッド）
- どの層も **「何を確認するか」を先に決めてから書く**
- テストの価値は、**実際にバグを捕まえるところ**にある
