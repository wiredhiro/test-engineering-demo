# 結合テストのサンプル：テスト専用DBを使う

単体テスト（`validateUsername`）に、**結合テスト**を1本足したもの。
「本物のDBを使って、部品のつなぎ目まで確認する」やり方を、実際に動くコードで示す。

対象：`registerUser(name, db)` … ユーザー登録。①バリデーション → ②DBで重複チェック → ③DBに保存、と複数の部品が連なる。

## 動かし方

```bash
npm install
npm test
```

→ 単体10件＋結合4件の、計14件が緑になる。

## 単体テストとの違い

| | 単体（validateUsername） | 結合（registerUser） |
|---|---|---|
| 調べる範囲 | 部品ひとつ（入力チェック） | ①〜③のつなぎ目 |
| DB | 使わない | **本物のDBを使う** |
| 例：確認できること | 「2文字は短すぎ」 | 「同じ名前は二重登録できない」 |

「二重登録できない」は、バリデーション単体では絶対に確認できない。
**"DBに既にある"という状態と、保存処理のつなぎ目**を見て初めて分かる。ここが結合テストの守備範囲。

## テスト専用DBの使い方（この結合テストの肝）

結合テストは、本番でも普段の開発用でもなく、**テスト専用の使い捨てDB**を使う。

- **本番DBは絶対に使わない** … テストはデータを消したり書いたりするので、本番が壊れる
- **普段の開発用DBも共用しない** … 残っているデータでテストが不安定になる（＝flaky）
- 代わりに、**テストごとに作り直せるメモリ上のSQLite（`:memory:`）**を使う

コードでは、この2つが要（前処理と後始末）：

```ts
beforeEach(() => {
  db = new Database(":memory:");           // 各テストの前に、まっさらな使い捨てDBを新しく作る
  db.exec("CREATE TABLE users (...)");     // 必要なテーブルを用意
});

afterEach(() => {
  db.close();                              // 後始末。次のテストにデータを残さない
});
```

これで **各テストが「空のDB」から始まる**（テスト1つ目でそれを確認している）。
どのテストから走っても結果が同じ＝順序に依存しない、安定したテストになる。

## 4つの結合テスト

| # | 確認していること | つなぎ目のどこを見るか |
|---|---|---|
| 1 | 各テストは空のDBから始まる | 使い捨てDBが毎回リセットされる |
| 2 | 新規登録が実際にDBに入る | ③保存が本当に効いたか（DB側から確認） |
| 3 | 同じ名前は二重登録できない | ②重複チェック → 保存を止める |
| 4 | バリデーションNGはDBに1件も入らない | ①で止まり、③まで進まない |

## Cloudflare D1 で使う場合

D1 は中身が SQLite なので、このサンプルの `:memory:` がそのまま考え方として効く。
Workers 全体を通しで試すなら、`wrangler` / Miniflare の**ローカル専用D1**（本番とは別の使い捨て）を使う。
どちらにせよ、テストで本番D1は触らせない。
