# ブランチ保護の設定メモ（CI を「必須」にする）

CI（[.github/workflows/test.yml](.github/workflows/test.yml)）は、置いただけでは「参考情報」でしかない。
**「緑じゃないと main にマージできない」**という門番にして、はじめて効く。
それを設定するのが GitHub の**ブランチ保護**。ここでは手順を残す。

---

## 前提：先に一度 CI を走らせておく

必須チェックとして選べるのは、**一度でも実行されたことのあるチェックだけ**。
だから順番は：

1. `test.yml` を含むコードを GitHub に push する
2. 適当なブランチで PR を1つ作り、CI（`static` / `test` / `e2e`）を走らせる
3. その後で、下のブランチ保護を設定する（チェック名が選択肢に出てくる）

チェック名は、ワークフローの**ジョブ名**そのまま：`static` / `test` / `e2e`。

---

## 手順（GitHub の画面）

1. リポジトリの **Settings（設定）** を開く
2. 左メニュー **Branches**（または **Rules → Rulesets**）を開く
3. **Add branch protection rule / Add ruleset** を押す
4. 対象ブランチに **`main`** を指定
5. 次の項目にチェックを入れる：

   | 設定項目 | 意味 |
   |---|---|
   | **Require a pull request before merging** | main への直接 push を禁止。必ず PR 経由に |
   | **Require status checks to pass before merging** | CI が緑じゃないとマージ不可 ← これが本命 |
   | └ 検索して `static` `test` `e2e` を選ぶ | この3つが全部緑になるまでマージできない |
   | **Require branches to be up to date before merging** | 最新の main に追従してから、が条件（任意・推奨） |
   | **Require conversation resolution before merging** | レビューコメントを全部解決してから（任意） |

6. （任意）**Do not allow bypassing the above settings** … 管理者も例外にしない。厳しめだが確実
7. **Create / Save** で保存

---

## 設定後にどうなるか

- main へ**直接 push できなくなる**（必ず PR を作る）
- PR 画面に CI の結果が並び、**`static` / `test` / `e2e` が全部緑になるまで「Merge」ボタンが押せない**
- 1つでも赤だと、原因（どのジョブ・どの観点）を直すまでマージできない

これで CI が「参考」から「門番」になる。**落ちるコードは main に入らない**、が仕組みで保証される。

---

## よくあるつまずき

- **チェック名が選択肢に出てこない** → まだ一度も CI が走っていない。先に PR を作って走らせる（前提を参照）
- **ジョブ名を変えたら必須チェックが外れた** → 必須チェックは「名前」で紐づく。`test.yml` のジョブ名を変えたら、ブランチ保護側も選び直す
- **E2E が重くてマージが待たされる** → E2E を必須から外し、「main マージ後」や「nightly」に回す手もある（全PR必須は `static` と `test` だけ、という段階運用）

---

## ひとことで

CI（自動テスト）＋ ブランチ保護（門番）は**セット**。
テストを書く → CI で自動実行 → ブランチ保護で「緑必須」にする、の3つがそろって、
はじめて「品質が仕組みで守られる」状態になる。
